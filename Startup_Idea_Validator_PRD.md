## 1. Overview

### 1.1 Problem statement
Aspiring founders decide whether to pursue a startup idea based on gut feel, because real validation — market sizing, competitive research, technical feasibility, and honest strategic risk assessment — takes hours of manual research most people skip.

### 1.2 Product summary
Startup Idea Validator is a web product that takes a plain-language startup idea and returns a decision-ready validation report — market opportunity (TAM/SAM/SOM), technical feasibility, SWOT analysis, and a numeric validation score with recommended next steps — generated by a multi-agent AI research crew (CrewAI) in 1–3 minutes, with live progress shown as each agent works.

### 1.3 Goals
- **Product goal:** give a founder a genuinely useful, honest go/no-go signal on an idea, backed by real web research and explicit math — not generic LLM encouragement.
- **Portfolio/engineering goal:** demonstrate a production-shaped async AI system — job queueing, live streaming progress, per-user auth and data isolation, and observability — not a synchronous script behind a form.

### 1.4 Non-goals (v1)
- Not a full business-plan generator or pitch-deck builder.
- Not a real-time chat/co-pilot for iterating on the idea (single request → single report, per run).
- No payments/billing in v1 (usage capped via rate limiting instead of monetized).
- No mobile app; responsive web only.

## 3. Functional requirements

### 3.1 Input handling
- FR-1: User must be authenticated (Supabase Auth) to submit an idea.
- FR-2: Accept free-text startup idea input (min length validation to avoid empty/garbage submissions; reasonable max length, e.g. 2,000 characters).
- FR-3: Enforce a per-user rate limit on submissions (e.g. N runs per day) to control LLM/API cost.

### 3.2 Multi-agent orchestration
- FR-4: On submission, create a `validation_runs` row (`status = pending`) and enqueue a background job — the request returns immediately with a `run_id`; it does not block on the AI work.
- FR-5: A worker process executes a 4-agent CrewAI crew against the submitted idea:
  1. **Market Researcher** — web search + market-size calculation (TAM/SAM/SOM)
  2. **Tech Analyst** — technical feasibility, stack, risk assessment
  3. **Business Strategist** — SWOT analysis (informed by market research output)
  4. **Report Compiler** — consolidates all outputs into the final report + score
- FR-6: As each agent starts/completes/calls a tool, the worker writes a `run_events` row (agent name, event type, message, timestamp).
- FR-7: The frontend subscribes to `run_events` for the active `run_id` via Supabase Realtime and renders live progress (no polling, no custom SSE server needed).
- FR-8: On completion, the worker updates `validation_runs` with `status = complete`, the final `report_markdown`, and `validation_score`. On failure, `status = failed` with an error message recorded in `run_events`.

### 3.3 Research & analysis tools
- FR-9: **Market Calculator** — a custom deterministic tool computing TAM/SAM/SOM from (population/unit count, addressable %, obtainable %, avg revenue per user), so market math is reproducible rather than LLM-estimated.
- FR-10: **Web search tool** (Serper or equivalent) available to Market Researcher and Tech Analyst for real, current information.
- FR-11: **SWOT Analyzer** — a custom tool that structures Strengths/Weaknesses/Opportunities/Threats input and flags any quadrant with fewer than 2 factors, forcing deeper analysis before finalizing.

### 3.4 Report generation
- FR-12: Final report is structured markdown with, in fixed order: Executive Summary, Market Opportunity, Technical Feasibility, Business Strategy (SWOT), Validation Score (1–10, integer, justified), Recommended Next Steps.
- FR-13: Report Compiler must be explicitly instructed to give an honest score — including low scores — rather than defaulting to encouraging language.
- FR-14: Report is persisted (not just displayed once) and viewable again later from history.

### 3.5 History & data access
- FR-15: Authenticated users can view a paginated list of their own past runs (idea text, score, date, status).
- FR-16: Users can open any past run to view its full report again.
- FR-17: Row Level Security enforced at the database level: a user can only read their own `validation_runs` and associated `run_events` — never another user's data, even via a misconfigured API bug.

### 3.6 Error handling
- FR-18: If a run fails (LLM error, tool failure, timeout), the user sees a clear failure state, not a silent hang — with the option to retry.
- FR-19: Runs stuck in `pending`/`running` beyond a max threshold (e.g. 10 minutes) are marked `failed` by a cleanup process, so the UI never shows an infinite spinner.

## 4. Non-functional requirements

| Category | Requirement |
|---|---|
| **Performance** | A validation run completes in under ~3 minutes for a typical idea; progress must visibly update at least once every ~15 seconds during a run (no long silent gaps) |
| **Scalability** | Celery workers scale horizontally independent of the API service; adding load should mean adding worker replicas, not redesigning the pipeline |
| **Security** | No secret keys (LLM, search, Supabase service role) ever reach the client; RLS enforced in Postgres, not just in application code |
| **Cost control** | Per-user daily run cap enforced before a job is enqueued, not after |
| **Reliability** | Failed runs are recorded with enough detail (`run_events`) to debug after the fact without reproducing live |
| **Observability** | Every agent step and tool call for every run is captured as data (`run_events`), not just logged to stdout and lost |

## 5. System architecture

```
┌───────────────┐        ┌──────────────────┐
│  React (Vite)   │◀──────▶│    Supabase        │
│  frontend        │  auth  │  - Auth             │
│                  │  +     │  - Postgres DB      │
│                  │realtime│  - Realtime engine   │
└───────┬─────────┘        └────────▲─────────┘
        │ POST /validate            │ writes rows
        ▼                           │
┌───────────────┐        ┌──────────┴─────────┐
│  FastAPI backend │──────▶│  Redis (queue)      │
│  (auth check +    │      └──────────┬─────────┘
│   enqueue only)   │                 │
└───────────────┘                 ▼
                          ┌──────────────────┐
                          │  Celery worker      │
                          │  runs CrewAI crew,  │
                          │  writes progress    │
                          │  to Supabase          │
                          └──────────────────┘
```

**Key architectural decisions:**
- **Async job model**: API never blocks on the AI work; it enqueues and returns a `run_id` immediately.
- **Supabase Realtime replaces custom SSE/websocket infra**: frontend subscribes directly to Postgres row changes on `run_events`.
- **Worker writes with the Supabase service role key** (server-side only, bypasses RLS by design — it's trusted infrastructure, not a user).
- **Frontend only ever holds the Supabase anon/public key**; data protection is enforced by RLS policies, not by hiding endpoints.

## 6. Data model

```sql
-- auth.users is managed by Supabase Auth

create table validation_runs (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references auth.users not null,
  startup_idea text not null,
  status text not null check (status in ('pending','running','complete','failed')),
  validation_score int,
  report_markdown text,
  created_at timestamptz default now(),
  completed_at timestamptz
);

create table run_events (
  id uuid primary key default gen_random_uuid(),
  run_id uuid references validation_runs not null,
  agent_name text,
  event_type text not null,   -- started | tool_call | completed | error
  message text,
  created_at timestamptz default now()
);

-- Row Level Security
alter table validation_runs enable row level security;
alter table run_events enable row level security;

create policy "users read own runs" on validation_runs
  for select using (auth.uid() = user_id);

create policy "users read own run events" on run_events
  for select using (
    run_id in (select id from validation_runs where user_id = auth.uid())
  );
```

## 7. API surface

| Endpoint | Method | Auth | Description |
|---|---|---|---|
| `/api/validate` | POST | required | Body: `{ startup_idea }`. Validates rate limit, inserts `pending` run, enqueues job, returns `{ run_id }` immediately |
| `/api/reports` | GET | required | Paginated list of the caller's own runs |
| `/api/reports/{run_id}` | GET | required (own run only) | Full run detail including report markdown |

Live progress is **not** a REST endpoint — the frontend subscribes directly to Supabase Realtime on `run_events` filtered by `run_id`.

## 9. Deployment plan

| Component | Platform | Notes |
|---|---|---|
| Frontend (React/Vite) | Vercel or Netlify | Static build, trivial CI deploy |
| Backend API | Railway (service 1) | Same Docker image as worker, different start command (`uvicorn`) |
| Worker | Railway (service 2) | Same image, `celery worker` start command; scales independently |
| Queue | Railway Redis add-on | Shared between API and worker |
| Database + Auth | Supabase (managed) | Free tier sufficient for v1 |
| Secrets | Railway env vars (API/worker only) | `OPENAI_API_KEY`/`ANTHROPIC_API_KEY`, `SERPER_API_KEY`, `SUPABASE_SERVICE_ROLE_KEY` — never shipped to the frontend |

**AWS alternative** (if AWS experience is specifically a portfolio goal): Fargate for API + worker services, ElastiCache for Redis, same architecture — more setup, no architectural difference.

## 11. Risks & mitigations

| Risk | Mitigation |
|---|---|
| LLM cost blowup from public/unlimited usage | Per-user daily rate limit enforced before enqueueing |
| Agent run hangs indefinitely | Max-duration cleanup job marks stuck runs as `failed` |
| Web search tool returns low-quality/irrelevant results, degrading report quality | Tool description constrains query behavior; Report Compiler instructed to flag low-confidence findings rather than presenting them as certain |
| Score feels artificially positive (LLM sycophancy) | Explicit prompt instruction to be honest/critical; consider spot-checking a sample of runs against manual judgment during Phase 1 |
| Supabase Realtime + RLS misconfiguration leaks data across users | RLS policies tested explicitly with two test accounts before Phase 2 sign-off |

## 12. Roadmap

**Phase 1 — Core async pipeline (no polished UI yet)**
FastAPI + Celery + Redis + CrewAI worker, Supabase schema + RLS, worker writes `run_events`/`validation_runs`, minimal frontend just to prove the Realtime subscription updates live. Success = a full run, end to end, with visible live progress.

**Phase 2 — Real frontend + auth**
React/Vite app, Supabase Auth (OAuth/magic link), history page, per-user rate limiting enforced.

**Phase 3 — Production polish**
Structured (`output_pydantic`) report format instead of raw markdown parsing, stuck-run cleanup job, cost logging per run, deployed live on Railway + Vercel + Supabase.

**Future (post-v1, out of scope now)**
Parallelize Market Researcher/Tech Analyst for latency; deeper observability (LangFuse tracing); side-by-side comparison of multiple validated ideas; export report to PDF.
